## Why this matters

`CLAUDE.md` (auto-generated by the `shadcn init` scaffold) documents an
atomic-design folder convention — `elements/` (atoms), `widgets/`
(molecules), `modules/` (organisms), `templates/` (layouts), `pages/` — that
this codebase does not use and, given its size (2 feature components:
`ListeningSpeakingApp`, `PhraseCard`), should not adopt: splitting a
430-line app into 5 taxonomy tiers would be pure ceremony with no
navigational benefit at this scale. Left as-is, `CLAUDE.md` is a stale
convention document — exactly the kind of drift the project's own audit
playbook flags as a finding in its own right. Since this repository is
explicitly set up for Claude Code (the file's existence proves that), any
future agent-driven change here will read `CLAUDE.md` first and may try to
honor instructions the codebase actively contradicts. This plan brings the
document in line with what the codebase actually does, without touching
any application code.

  1. Directories in CamelCase, with only index.tsx file

  4. The component should be in the folder `components`

  6. Sub-components do not need to repeat the parent prefix in their name
  7. Add an `id` attribute to every component to make them easy to locate and test
